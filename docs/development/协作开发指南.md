# DeepSecurity 协作开发指南

本文档用于团队成员开发前快速对齐流程、目录职责、提交规范和安全要求。项目当前是 Flask 后端 + Vue/Vite 前端 + 安全分析业务模块的组合仓库，开发时请尽量保持改动边界清晰。

## 1. 核心原则

1. **主分支保持稳定。** 不直接把未验证代码推到 `master`，功能、修复和文档改动先走独立分支，再通过 Pull Request 合并。
2. **一次一分支。** 一个分支只解决一个明确问题，例如一个页面、一个接口、一个检测规则或一份文档。
3. **接口先行。** 前后端联调前先约定接口路径、请求参数、响应字段和错误格式，再分别实现。
4. **不提交敏感信息。** `.env`、API Key、数据库密码、Neo4j 密码、私钥、真实内网地址和抓包上传数据都不能提交。
5. **提交前必须能说明验证结果。** 至少说明跑过哪些测试、构建或手工验证，不能只写“应该没问题”。

## 2. 项目目录分工

```text
frontend/                 Vue 3 + Vite 前端
frontend/src/api/         前端 API 封装，接口路径变更要同步这里
frontend/src/views/       页面组件
xiaoxueqi/                Flask 应用、蓝图路由和传统模板
utils/                    检测、捕获、溯源、归因、日志、流量等业务模块
knowledge_base/           ATT&CK、APT、RAG 知识库和建库脚本
docs/                     项目文档、实验文档、开发协作文档
tests/                    自动化验证测试
data/                     运行时数据，默认不提交
```

常见改动归属：

- 新页面或前端交互：改 `frontend/src/views/`，必要时同步 `frontend/src/api/` 和 `frontend/src/router/`。
- 后端接口：改 `xiaoxueqi/views/`，业务逻辑尽量放到 `utils/` 对应模块。
- 检测、溯源、归因算法：优先改 `utils/detection/`、`utils/trace/`、`utils/attribution/`。
- 数据库结构或初始化脚本：改 `knowledge_base/create_database.sql`，并在文档说明迁移方式。
- 使用说明、部署说明、协作规范：放到 `docs/` 或更新 `README.md`。

## 3. 开发流程

### 第一步：同步主分支

```powershell
git checkout master
git pull origin master
```

### 第二步：创建开发分支

分支命名建议使用 `类型/简短描述`：

```text
feat/detection-filter
fix/traceback-report-empty
docs/development-guide
refactor/traffic-parser
chore/update-deps
```

```powershell
git checkout -b docs/development-guide
```

### 第三步：配置本地环境

后端：

```powershell
.\venv\Scripts\Activate.ps1
pip install -r requirements.txt
```

前端：

```powershell
cd frontend
npm install
```

敏感配置参考 `.env.example`，在本机环境变量里设置，不要把真实值写进仓库。

### 第四步：开发和自测

后端基础验证：

```powershell
python -m unittest tests.test_config_security -v
python -m compileall config.py utils xiaoxueqi
```

前端验证：

```powershell
cd frontend
npm run build
```

如果只改文档，也至少检查 Markdown 链接、路径和命令是否与当前项目一致。

### 第五步：提交代码

提交信息使用 `type: description`：

```text
feat: add detection severity filter
fix: handle empty traceback report
docs: add collaboration guide
refactor: split traffic parser helpers
chore: update gitignore rules
```

```powershell
git status --short
git add <changed-files>
git commit -m "docs: add collaboration guide"
```

### 第六步：推送分支并发起 PR

```powershell
git push origin docs/development-guide
```

在 GitHub 上创建 Pull Request，标题写清楚做了什么，正文至少包含：

- 改动摘要
- 影响范围
- 验证结果
- 是否涉及配置、数据库、密钥或部署步骤

### 第七步：代码审查和合并

- 审查人发现问题时选择 Request changes，并给出具体文件和修改建议。
- 作者修改后继续推到同一分支，PR 会自动更新。
- 至少一名审查人 Approve 后再合并。
- 合并时推荐使用 Squash and merge，保持 `master` 历史清晰。
- 合并后删除远程分支。

## 4. 接口协作规范

新增或修改接口时，先在 PR 描述或文档中说明：

```text
Method: GET
Path: /detection/api/alerts
Query: severity, page, limit
Response:
{
  "ok": true,
  "items": [],
  "total": 0
}
Error:
{
  "ok": false,
  "error": "message"
}
```

后端实现接口后，前端在 `frontend/src/api/` 增加或更新封装，页面组件不要到处手写重复请求路径。

## 5. 安全要求

提交前必须确认：

- `.env`、私钥、真实 API Key 没有进入暂存区。
- 数据库密码、Neo4j 密码、云服务 Token 不写死在代码、文档或截图里。
- `data/uploads/`、`data/live_captures/`、`frontend/node_modules/`、`xiaoxueqi/static/spa/` 不提交。
- 日志输出不打印密钥、密码、完整连接串或敏感主机信息。
- 示例命令使用 `<SERVER_IP>`、`<DB_PASSWORD>`、`localhost` 等占位值。

推荐提交前检查：

```powershell
git status --short
git diff --cached --name-only
rg -n "sk-[A-Za-z0-9]{20,}" . --glob "!frontend/node_modules/**" --glob "!data/**" --glob "!.git/**"
```

## 6. 冲突处理

如果推送或合并时遇到冲突：

```powershell
git checkout <your-branch>
git pull origin master
```

打开冲突文件，搜索 `<<<<<<<`，手动保留正确内容。处理完后：

```powershell
git add <resolved-files>
git commit -m "fix: resolve merge conflicts"
git push origin <your-branch>
```

不要用强制覆盖的方式解决自己不理解的冲突。

## 7. Reviewer 检查清单

审查前确认：

- [ ] 改动范围是否和 PR 标题一致。
- [ ] 是否误提交了依赖目录、缓存、抓包、构建产物或本地配置。
- [ ] 是否有硬编码密钥、密码、真实内网地址或敏感日志。
- [ ] 前后端接口字段是否一致。
- [ ] 数据库或配置变更是否写清楚迁移/设置方式。
- [ ] 作者是否提供了测试、构建或手工验证结果。

## 8. 常见问题

### npm install 或 npm run build 失败

先确认在 `frontend/` 目录内执行命令。如果依赖状态异常，删除 `frontend/node_modules/` 后重新运行 `npm install`。

### 后端连接数据库失败

先检查本机环境变量是否按 `.env.example` 设置。没有数据库时，部分页面可以使用内存模拟数据或场景数据，但真实 SQL Server/Neo4j 流程会失败。

### LLM 功能不可用

确认 `LLM_API_KEY` 和 `LLM_BASE_URL` 是否设置。未配置时，规则检测仍可运行，LLM 报告和深度分析会跳过或返回未配置提示。

### 不确定文件该放哪里

先按目录职责选择最接近的位置；如果同时涉及前端、后端和算法，PR 描述中明确每部分改动，避免把无关重构混进同一个分支。
